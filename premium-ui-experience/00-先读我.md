# 高质量 UI 体验 skill

> 目的：让 AI 不只是把页面“做出来”，而是更稳定地把页面做得 **更像成熟产品**。  
> 用法：把整个文件夹放进项目里，让 AI 先读 `SKILL.md`；如果项目里有 `design.md`，再让 AI 先读 `design.md` 后再动手。

---

## 1. 这个 skill 解决什么问题

它主要解决 4 类问题：

- 页面功能做完了，但视觉很普通
- 界面能用，但交互不顺、反馈不明确
- 动画加了，但显得乱或拖慢体验
- 每轮都说“做高级一点”，但 AI 输出不稳定

这个 skill 的核心目标是让 AI 在项目里形成相对稳定的高质量 UI / UX 工作方式。

---

## 2. 最推荐的用法

### 场景 A：新项目准备建立视觉基线

让 AI 先读：

- `SKILL.md`
- `references/02-design.md使用规范.md`

如果项目里还没有 `design.md`，先用脚本初始化一份。

### 场景 B：已有页面，但想整体提升质感

让 AI 先读：

- `SKILL.md`
- 项目里的 `design.md`
- 当前页面 / 组件文件

然后先输出：

- 本轮目标视觉气质
- 页面结构问题
- 主要交互问题
- 计划提升点

再开始改代码。

### 场景 C：已经有功能，只想做最后的 UI 打磨

优先关注：

- 布局节奏
- 排版层级
- 配色与对比
- 组件状态
- hover / focus / loading / empty / error
- 微交互与性能

---

## 3. 最关键的一点

### 不要只靠一句“做得更好看一点”。

更可靠的做法是：

- 用这个 skill 规定设计工作流
- 用项目里的 `design.md` 固定项目自己的审美与交互基线

也就是说：

- **skill**：负责方法、步骤、检查方式
- **`design.md`**：负责项目自己的风格、气质、参考、偏好与禁忌

---

## 4. 这个 skill 最适合和什么一起用

建议这样搭配：

- `01-开发约束skill/`：先收敛范围，避免功能过重
- `03-高质量UI体验skill/`：再提高界面和交互质量
- `项目上下文守护skill/`：在开发过程中持续记住项目状态、规则和最近进展

如果当前环境支持额外设计技能，也可以按需叠加：

- `frontend-design`
- `arrange`
- `typeset`
- `colorize`
- `animate`
- `polish`
- `harden`
- `optimize`
- `audit`

---

## 5. 最后一句提醒

**高质量 UI 不是“加更多装饰”，而是把结构、信息层级、组件状态、交互反馈、动效节奏和视觉语言一起做对。**
