# 04-Token治理与上下文经济学skill

这个 skill 不是为了把回答变短，而是为了让 AI 在真实开发中 **少读无关内容、少做无效动作、少产生返工**。

## 这个 skill 解决什么问题

很多项目进入中后期后，AI 的 token 消耗会失控，通常不是因为任务真的太难，而是因为出现了这些问题：

- 一上来就读太多文件
- 反复重读同一批上下文
- 改动范围越来越大
- 中间汇报越来越长
- 没有先定位就开始大改
- 没有验证就先输出大量解释

这个 skill 的作用，就是把这些行为改成更像成熟工程团队的执行方式。

## 最推荐的使用方式

建议把它放在这条链路里：

1. 先用 `01-开发约束skill/` 收敛边界和最小闭环
2. 再用 `04-Token治理与上下文经济学skill/` 控制上下文读取、改动半径和验证成本
3. 如果任务涉及界面，再叠加 `03-高质量UI体验skill/`
4. 收尾时用 `项目上下文守护skill/` 压缩沉淀长期记忆

## 它和“项目上下文守护”有什么区别

- `项目上下文守护skill/`：负责 **把该记住的东西沉淀下来**
- 这个 skill：负责 **在当前轮里只读必要信息，不把上下文用爆**

一个偏“长期记忆”，一个偏“当前执行成本控制”。

## 它和“开发约束”有什么区别

- `01-开发约束skill/`：先决定 **该不该做、做多大、怎么收口**
- 这个 skill：再决定 **怎么用最少成本做这件事**

一个偏“任务治理”，一个偏“上下文与执行治理”。

## 建议配套文件

建议在具体项目里维护一个 `execution-budget.md`，明确：

- 先读哪些文件
- 默认不要全量读哪些目录
- 常见任务从哪里开始搜索
- 什么改动跑什么验证
- 什么检查过重，不要默认执行

如果项目里还没有，可以用 `scripts/init_execution_budget_md.py` 初始化。

## 一句话理解

这个 skill 要建立的不是“回答更短”，而是：

**先搜索、后阅读；先收口、后扩圈；先验证、后解释。**
