# 输出压缩与汇报规范

## 目标

减少无效解释，不牺牲透明度。

## 对用户真正有价值的内容

默认只保留：

- 关键判断
- 关键依据
- 关键改动
- 关键验证
- 关键风险

## 中间汇报规则

中间汇报应当：

- 短
- 真
- 只说新增信息
- 只说下一步最关键动作

不要把每一次搜索、每一个读取动作都展开成作文。

## 结果总结规则

结果总结优先回答：

1. 改了什么
2. 为什么这么改
3. 怎么验证的
4. 还有什么未处理

如果用户没有要求，不必重复贴代码或长篇复述文件内容。

## 压缩原则

### 1. 不复述用户已知信息

用户刚说过的目标，不要原样再说一遍。

### 2. 不复述已经落地在文件里的内容

只提最重要变化，不重贴整份文档。

### 3. 不把过程当成果

搜索过、想过、比较过，不等于完成了价值。

### 4. 不把“详尽”误当“专业”

啰嗦常常只是上下文泄漏，不是专业度。

## 允许详细的情况

只有以下情况，才主动展开：

- 用户明确要求详细过程
- 任务本身是调研、审计、评估
- 需要展示关键代码证据
- 需要说明一个不直观的权衡

## 一句话原则

**让输出像交接结论，不像思维草稿。**
