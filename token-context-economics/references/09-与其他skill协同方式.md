# 与其他 skill 协同方式

## 目标

让这个 skill 成为“执行成本控制层”，而不是和其他 skill 职责重叠。

## 推荐协同顺序

### 1. `01-开发约束skill/`

先回答：

- 该不该做
- 做多大
- 第一轮最小闭环是什么
- 本轮不做什么

### 2. `04-Token治理与上下文经济学skill/`

再回答：

- 该读哪些信息
- 从哪里开始搜索
- 本轮改动半径多大合适
- 本轮验证跑到哪一层

### 3. `03-高质量UI体验skill/`

如果任务涉及 UI，再回答：

- 如何确保视觉和交互质量
- 如何去 AI 味、避免随意发挥
- 如何依照 `design.md` 做高质量实现

### 4. `项目上下文守护skill/`

收尾时再回答：

- 哪些内容值得沉淀为长期记忆
- 当前状态快照怎么压缩
- 哪些经验或规则要进入共享上下文

## 典型组合

### 组合 1：新功能但范围容易膨胀

推荐：

- `01-开发约束skill/`
- `04-Token治理与上下文经济学skill/`
- `项目上下文守护skill/`

### 组合 2：高要求 UI 页面迭代

推荐：

- `01-开发约束skill/`
- `04-Token治理与上下文经济学skill/`
- `03-高质量UI体验skill/`
- `项目上下文守护skill/`

### 组合 3：局部 bug 修复

推荐：

- 小范围时直接使用 `04-Token治理与上下文经济学skill/`
- 如果边界不清，再补 `01-开发约束skill/`

## 不建议的误用

- 不要让 `项目上下文守护skill/` 承担当前轮的 token 控制职责
- 不要让 `03-高质量UI体验skill/` 替代范围治理和验证预算判断
- 不要把 `01-开发约束skill/` 当成 every step 都要完整重跑的前置流程
