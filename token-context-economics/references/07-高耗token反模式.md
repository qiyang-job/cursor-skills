# 高耗 token 反模式

## 反模式 1：全量阅读冲动

表现：

- 一上来就读整个 `README`
- 顺手把整个 `docs/` 和 `.ai-context/` 都扫一遍
- 为了“保险”把所有候选文件都打开

后果：

- token 快速膨胀
- 真正关键线索被噪音淹没

## 反模式 2：重复读取

表现：

- 同一轮里多次重读同一个文件
- 已确认稳定的规则文件反复打开
- 每次新任务都从头重读同样背景

后果：

- 成本增加
- 认知焦点被旧信息干扰

## 反模式 3：未定位先大改

表现：

- 还没确认入口就开始设计方案
- 还没确认根因就开始重构

后果：

- 返工概率极高
- 验证范围被迫扩大

## 反模式 4：顺手扩圈

表现：

- 修一个问题，顺手改三个模块
- 做一个页面，顺手统一全站风格
- 改一个接口，顺手重整 API 层

后果：

- 改动半径飙升
- 验证成本跟着上涨

## 反模式 5：验证失衡

表现：

- 小改动跑整仓重型检查
- 大改动只做表层 smoke

后果：

- 要么浪费，要么返工

## 反模式 6：中间汇报过载

表现：

- 每几步都写很长状态说明
- 把思考过程完整暴露给用户
- 重复总结已经确认的信息

后果：

- token 花在沟通表演上，而不是问题推进上

## 反模式 7：把长期记忆写成流水账

表现：

- 所有过程都记进 handoff
- 所有变化都塞进 current context
- 不做归档和压缩

后果：

- 下一轮加载成本继续抬高

## 反模式 8：为了省 token 而省掉验证

表现：

- 不跑检查
- 不确认入口
- 不看真实结果

后果：

- 省的是眼前 token，亏的是后续多轮返工
