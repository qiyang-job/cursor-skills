# 最小改动半径规则

## 目标

把每次改动控制在一个可验证、可回退、可解释的最小闭环里。

## 为什么这和 token 有关

改动半径一旦失控，token 消耗会同时从 3 个方向增长：

- 需要读更多文件
- 需要解释更多变更
- 需要跑更多验证并处理更多返工

## 默认闭环定义

一次健康的改动应尽量满足：

- 只解决一个核心问题
- 只引入一组主要规则变化
- 只动最少必要文件
- 可以用一轮局部验证确认结果

## 开工前先回答 4 个问题

1. 这次真正要解决的一个问题是什么
2. 哪几个文件是本轮必须改的
3. 哪些问题虽然看见了，但先不处理
4. 什么行为会让本轮范围突然变大

## 高风险扩圈信号

出现以下信号时，默认暂停并重新评估：

- 从一个组件改到多个页面
- 从一个页面改到共享状态或全局配置
- 从一个 bug 修复演变成结构重构
- 从功能实现演变成设计系统重定义
- 从本地逻辑改动演变成 API 协议调整

## 处理策略

### 如果是必要扩圈

显式说明：

- 为什么原闭环不够
- 现在最小的新闭环是什么
- 新增了哪些验证需求

### 如果只是顺手优化冲动

默认不做，记入：

- handoff
- lessons learned
- follow-up

不要把“顺手”当成零成本。

## 一句话原则

**优先做最小可验证修复，而不是最完整的一次性理想改造。**
