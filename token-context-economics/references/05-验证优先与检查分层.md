# 验证优先与检查分层

## 目标

让验证强度与改动风险匹配，既避免漏检，也避免一刀切地过度检查。

## 核心原则

- 先定义验收口径，再选验证方式
- 先做最贴近改动的验证，再决定是否扩展
- 先拿到真实结果，再写解释

## 分层验证模型

### 第 1 层：语法 / 类型 / 构建前检查

适用于：

- TypeScript / Python / 配置文件修改
- 明显可能引入语法问题的改动

### 第 2 层：局部 lint

适用于：

- 改过的文件
- 相关目录
- 已知高约束模块

### 第 3 层：目标功能验证

适用于：

- bug 修复
- 局部功能新增
- UI 状态修正

优先验证本次真正改的主链路。

### 第 4 层：扩大验证

仅在这些情况下加大范围：

- 改动涉及共享逻辑
- 改动涉及公共组件或基础设施
- 局部验证结果异常
- 用户明确要求完整回归

## 常见误区

### 误区 1：小改动也默认全量跑

这会显著抬高成本，并延长反馈周期。

### 误区 2：高风险改动只做表层验证

这会在后续返工中消耗更多 token。

### 误区 3：没验证就先总结

这会把“猜测性结果”包装成“完成状态”。

## 推荐输出格式

验证相关输出尽量压缩为：

- 跑了什么
- 为什么跑这个
- 结果怎样
- 还剩什么没覆盖

不要把整段日志无差别贴给用户，除非它本身就是问题核心。
