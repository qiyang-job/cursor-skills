# 效果评估与验收

## 目标

判断这个 skill 是否真的让开发更高效，而不是只是让输出更短。

## 核心验收维度

### 1. 定位效率

期望变化：

- 更快找到真正相关的入口文件
- 更少出现“读了一圈还没开始动手”的情况

### 2. 上下文利用率

期望变化：

- 平均读取文件数下降
- 重复读取率下降
- 无关文档加载减少

### 3. 改动收口能力

期望变化：

- 单次改动文件数更可控
- 顺手扩圈变少
- 返工轮数下降

### 4. 验证匹配度

期望变化：

- 小改动不再默认跑过重检查
- 高风险改动不再只做表层验证

### 5. 汇报质量

期望变化：

- 中间汇报更短
- 结果总结更聚焦
- 复述性内容明显减少

## 建议观察指标

可以用以下指标做简单复盘：

- 单次任务平均读取文件数
- 单次任务重复读取同一文件的次数
- 单次任务工具调用数
- 单次任务修改文件数
- 单次任务验证命令数
- 返工轮数
- 用户追问“所以你到底改了什么”的次数

## 合格标准

如果连续几轮真实任务中满足以下多数项，可以认为 skill 已开始发挥作用：

- 能较稳定地从搜索而不是全量阅读起步
- 能控制在最小必要文件集内完成局部任务
- 输出明显更聚焦，但结果仍准确
- 验证更贴合本次风险
- 没有因为省 token 而显著增加返工

## 不合格信号

出现以下情况，说明 skill 需要继续迭代：

- 虽然回答短了，但误改变多
- 虽然读取少了，但经常漏掉关键依赖
- 虽然汇报短了，但用户必须反复追问细节
- 虽然检查少了，但后续返工明显增加

## 一句话结论

这个 skill 的正确目标是：

**用更少的上下文和更少的返工，换到同等或更高的交付质量。**
