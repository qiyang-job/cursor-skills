# 搜索优先与最小阅读

## 核心原则

默认先搜索，再读文件。

很多高耗 token 问题不是来自实现本身，而是来自：

- 还没定位到入口就开始通读文件
- 明明只需要一个符号定义，却把整个模块都读了
- 明明只需要两个候选文件，却把十几个文件都打开了

## 典型顺序

### 场景 1：找实现入口

优先做：

- 语义搜索
- 精确搜索函数名、路由名、配置键
- 文件名搜索

再读：

- 搜索命中的最强候选
- 候选附近的少量上下文

### 场景 2：找报错来源

优先做：

- 搜索报错文案
- 搜索日志前缀
- 搜索异常类型
- 搜索最近相关接口或状态名

再读：

- 抛错点
- 调用点
- 相关校验逻辑

### 场景 3：加一个局部功能

优先做：

- 找现有相似功能
- 找复用组件或相近页面
- 找已有测试和样式模式

再读：

- 当前改动目标
- 最接近的复用参考

## 最小阅读规则

默认遵守：

- 先读一个候选文件，不够再加第二个
- 第二个文件优先选择直接依赖方，而不是随机扩散
- 尽量围绕符号、路由、组件、接口、状态名来读
- 不因为“也许有用”就把周边模块全读完

## 什么时候允许读得更重

只有以下情况，才允许显著扩大阅读量：

- 用户明确要求系统梳理
- 当前问题涉及复杂跨模块行为
- 需要确认多个入口的一致性
- 项目存在明显半迁移状态，局部证据不可信

## 好的行为示例

- 先定位到 `route.ts` 再读实现，而不是先看整个 `api/` 目录
- 先找相似组件再改 UI，而不是从零重读整个设计系统
- 先找调用链上的直接三四个文件，而不是整页整模块横向展开

## 坏的行为示例

- 一上来读完整个 `README`、`docs` 和 `src`
- 搜到一个命中后，又为了“保险”把所有命中全读一遍
- 同一轮里多次重读已经确认无关的文件
