# GitHub 最佳实践对齐评估

## 结论

这版 `04-Token治理与上下文经济学skill` 已经和 GitHub 公开实践中的几条核心路线基本对齐：

- **稳定入口而非到处猜**
- **模块化上下文而非全量灌入**
- **搜索 / 地图优先而非全量阅读源码**
- **上下文压缩而非过程复读**
- **验证门禁而非只靠长解释**

如果按成熟度打分，我会给当前版本一个 **`8/10`**：

- 方向已经对
- 结构已经完整
- 可以投入真实项目使用
- 但还没有做成带自动统计和自动化 repo map 的重基础设施版本

## 这次对照了哪些公开样本

### 1. `AGENTS.md` 规范

对照来源：

- `agents.md`
- GitHub 官方文章《How to write a great agents.md: Lessons from over 2,500 repositories》

对齐点：

- 给 agent 一个稳定、可预测的上下文入口
- 让规则文件短小、任务相关、易于扫描
- 明确边界、命令、目录和禁止项
- 减少无关背景加载，避免“通用助手化”

本 skill 的对应设计：

- 强调先读最近、最相关的规则和状态文件
- 建议项目维护 `execution-budget.md`
- 强调只读当前任务所需上下文，而不是重复扫全局背景

### 2. GitHub 官方的 context engineering / agentic workflows

对照来源：

- GitHub Blog: `How to build reliable AI workflows with agentic primitives and context engineering`

对齐点：

- 结构化执行，而不是临场发挥
- 模块化规则和记忆
- 会话与上下文按任务分层
- 在工作流中嵌入验证门禁

本 skill 的对应设计：

- 把执行过程拆成任务分类、最小上下文包、搜索、改动半径、验证、压缩收尾
- 强调验证优先与检查分层
- 明确哪些信息要进入长期记忆，哪些只属于本轮过程

### 3. Aider 的 `RepoMap`

对照来源：

- `Aider-AI/aider` 仓库中的 `repomap.md`

对齐点：

- 用代码地图和关键符号概览代替全量源码
- 在 token 预算内，只挑最相关的上下文
- 根据当前聊天状态动态调整上下文大小

本 skill 的对应设计：

- 强调先搜索、后读取
- 强调只读最小候选文件集
- 强调围绕入口、符号、调用链来选文件，而不是整仓扫读

### 4. `awesome-context-engineering`

对照来源：

- `yzfly/awesome-context-engineering`

对齐点：

- 最小上下文
- 检索优先
- 分层记忆
- 上下文压缩
- 工具优先

本 skill 的对应设计：

- 在 `02-上下文分层读取策略.md` 中做了读取层级划分
- 在 `03-搜索优先与最小阅读.md` 中强调检索优先
- 在 `06-输出压缩与汇报规范.md` 中强调压缩而不是复读

### 5. `code-review-graph`

对照来源：

- `tirth8205/code-review-graph`

对齐点：

- 持久化代码图谱
- 爆炸半径分析
- 只读最小相关文件集
- 用更紧凑的结构化上下文替代全量源码

本 skill 的对应设计：

- 在 `04-最小改动半径规则.md` 中强调改动半径
- 在 `05-验证优先与检查分层.md` 中强调验证要跟随风险与影响面
- 在整体工作流中强调最小上下文包和最小安全改动闭环

## 当前版本的优势

- 已经把“省 token”从技巧层，提升到工作流层
- 已经把“少读、少改、少返工、少复读”写成硬规则
- 已经和现有 3 个 skill 分工清楚，不是职责重叠
- 已经有项目级模板 `execution-budget.md` 作为落地抓手

## 当前版本还缺什么

### 1. 自动化度还不够高

现在主要是流程和规范层，尚未加入：

- 自动统计重复读取率
- 自动计算常见任务的读取预算
- 自动生成 repo map 或最小文件集

### 2. 还没有长期实战统计闭环

目前已经有结构对齐和方法对齐，但还没有在多个真实项目里持续记录：

- 平均读取文件数变化
- 平均返工轮数变化
- 平均工具调用数变化

### 3. 还可以再补一个更工程化的 v2

后续可升级方向：

- 增加本地 repo map / symbol map 生成脚本
- 增加任务后复盘脚本，统计高耗 token 行为
- 增加针对常见任务类型的预算模板

## 总结

这版 skill 已经不只是“少说话”的建议，而是基本对齐了 GitHub 公开实践中的这条主线：

**用稳定入口、模块化上下文、检索优先、压缩输出和验证门禁，换来更低成本但更可靠的执行。**
