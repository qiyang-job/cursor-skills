# Token 治理目标与工作方式

## 目标

Token 治理不是单纯追求“更短”，而是追求 **单位 token 的有效产出更高**。

一个健康的工作状态应当表现为：

- 找问题更快
- 误改更少
- 返工更少
- 汇报更聚焦
- 长项目里不容易上下文失控

## 关注的不是节省，而是浪费

优先减少这些浪费：

- 读了很多，但对决策没有帮助
- 读过一次，又重复读取
- 没定位清楚就开始改，导致返工
- 运行明显过重的验证
- 输出大量过程描述，但没有新增有效信息

## 工作方式

### 1. 先判断任务大小

先判断是：

- 小修
- 局部迭代
- 中等复杂度改动
- 复杂探索
- 全局评估

不同任务大小，对应不同的读取与验证预算。

### 2. 先拿到最小事实集

默认先收集最少事实：

- 用户要什么
- 项目已有硬规则是什么
- 当前状态是什么
- 入口代码在哪里
- 本轮最可能要改哪几处

### 3. 再决定是否扩圈

只有以下情况才扩大读取范围：

- 搜索结果互相矛盾
- 当前文件无法解释行为来源
- 需要确认调用链或共享状态
- 需要跑更强验证才能判断风险

### 4. 把验证纳入预算

验证不是越多越好，而是越匹配风险越好。

### 5. 收尾时压缩

最后只保留能支持后续行动的信息，不保留大段过程性废话。

## 判断这个 skill 是否运作正确

不是看字数少没少，而是看：

- 一次任务是否更快收敛到正确文件
- 是否减少重复读取
- 是否减少返工轮数
- 是否减少“看起来很认真、实际上没前进”的长汇报
