# 反模式与遗忘风险

## 1. 为什么 AI 会“失忆”

很多时候，不是 AI 突然变笨了，而是项目上下文没有被持续沉淀。

高频原因包括：

- 重要背景只存在于聊天里
- 决策做了，但没有落到文档
- 当前状态没有被更新
- 每轮都只关注眼前任务，没有补项目全貌
- 会话过长，但没有定期压缩摘要

---

## 2. 常见反模式

### 反模式 1：把所有信息都放在一个超长文档里

结果：谁都懒得读，真正关键的内容被埋掉。

### 反模式 2：只靠 README 承担全部上下文

结果：README 通常只适合高层说明，不适合承载当前状态、交接和决策日志。

### 反模式 3：只靠聊天，不落记忆文件

结果：这轮聊得很清楚，下一轮还是会忘。

### 反模式 4：想逐字保存每轮完整对话

结果：噪音膨胀，读取成本变高，真正有效信息反而难找。

### 反模式 5：做完改动不更新状态

结果：代码已经变化，但记忆仍停在旧状态，AI 反而更容易判断错。

---

## 3. 更可靠的做法

更好的方法是：

- 用多个小文件分工保存不同类型信息
- 在关键节点持续更新，而不是拖到最后一次性补
- 用摘要和决策记录代替全文转录
- 恢复开发前先同步项目快照，再动手改代码

---

## 4. 一个关键判断标准

### 如果记忆文件让“恢复开发”更快，那它就是好设计；如果只是让文档更多，那它就是坏设计。
