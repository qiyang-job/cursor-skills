# GitHub 最佳实践对齐评估

## 1. 先说结论

当前这套“项目上下文守护 skill” **已经对齐了 GitHub 上公开常见实践中的核心思想**，尤其是：

- 用项目本地文件沉淀共享记忆，而不是只靠聊天
- 区分稳定事实、当前状态、决策、交接、未决问题、变更记录
- 强调摘要压缩，而不是保存原始完整对话
- 让下一轮 / 下一位 AI / 下一位开发者能快速恢复上下文

但它目前更适合定义为：

> **结构方向正确、可落地使用的 v1 版**

而不是：

> **已经在多项目、多轮长会话、多 agent 环境里充分验证过效果的成熟版**

---

## 2. 我本次对照了哪些公开实践

本次主要对照了几类 GitHub 上公开可见的上下文 / handoff / 记忆体系：

### 2.1 AAHP（AI-to-AI Handoff Protocol）

AAHP 的代表特点：

- 在项目里建立 `.ai/handoff/`
- 通过 `MANIFEST.json` 做索引和快速上下文入口
- 用 `STATUS.md`、`NEXT_ACTIONS.md`、`LOG.md` 等分工保存状态
- 强调 **压缩后的结构化状态**，而不是原始聊天历史
- 限制日志长度、限制活跃任务数、支持归档

### 2.2 ADR（Architectural Decision Records）

ADR 的代表特点：

- 用结构化文档记录关键决策
- 明确记录 **为什么这样做**
- 记录替代方案与权衡
- 形成可长期维护的决策日志

### 2.3 Context Handoff Engine

这类实践的代表特点：

- 用 `MEMORY.md` 做索引文件
- 详细内容拆到多个 topic files 中按需读取
- 用 handoff 文件支持跨会话恢复
- 记录 lessons / mistakes，避免重复犯同样的问题
- 有明确的更新和消费机制

### 2.4 OpenClaw Memory / Context Architecture

这类实践的代表特点：

- 用分层记忆，而不是单一大文档
- 有固定唤醒顺序
- 用压缩后的 `CURRENT_CONTEXT` 做快速恢复
- 强调 verification-first（没有验证过就不要算完成）
- 强调 failure-to-improvement loop（失败要沉淀成资产）

---

## 3. 当前 skill 已经对齐的部分

### 3.1 共享记忆落在项目本地

当前 skill 采用 `.ai-context/` 目录保存项目记忆，这一点和 GitHub 上较成熟实践 **高度一致**。

它解决的是“聊天窗口会失忆，但项目文件不会”的问题。

### 3.2 分文件分工是正确方向

当前这 7 个文件：

- `00-project-overview.md`
- `01-architecture-map.md`
- `02-current-state.md`
- `03-decision-log.md`
- `04-session-handoff.md`
- `05-open-questions.md`
- `06-change-log.md`

与公开最佳实践的核心分层是匹配的：

- **项目目标 / 边界**
- **架构全貌**
- **当前状态**
- **决策日志**
- **交接摘要**
- **未决问题**
- **重要变更**

这说明基础结构没有跑偏。

### 3.3 决策日志方向正确

`03-decision-log.md` 的设计与 ADR 的核心思想是对齐的：

- 记录决策内容
- 记录原因
- 记录替代方案
- 记录影响范围

这点很重要，因为“决策为什么这样定”是最容易在多轮对话中被遗忘的内容之一。

### 3.4 交接文件定位正确

`04-session-handoff.md` 被定义成“下一位继续的人最该先读的一页纸”，这个方向和 handoff engine、AAHP 一类协议的意图是一致的。

### 3.5 明确反对保存完整聊天

当前 skill 明确反对：

- 逐字保存每轮对话
- 把所有信息塞进一个超长文件
- 用聊天流水账替代项目状态

这与 GitHub 上较成熟的上下文工程实践 **高度一致**。

---

## 4. 当前 skill 还没完全对齐的地方

### 4.1 缺少一个“超轻入口索引”

公开实践里一个很常见的优化是：

- 先读一个很小的入口文件
- 再决定要不要读细节文件

比如：

- `MANIFEST.json`
- `MEMORY.md`
- `CURRENT_CONTEXT.md`

而当前 skill 只有固定读取顺序，没有一个真正意义上的：

> **低成本、极短、用于快速唤醒的总入口**

这意味着它在结构上是对的，但在“冷启动成本”上还可以更进一步。

### 4.2 缺少“压缩快照层”

现在的 `04-session-handoff.md` 承担了一部分“快照”职责，但它还不等于一个专门的：

- 当前项目一句话状态
- 当前主任务
- 当前阻塞
- 下一步
- 必读文件列表

如果有一个专门的 `CURRENT_CONTEXT.md` 或 `00-quick-context.md`，恢复速度会更稳。

### 4.3 缺少约定 / 规则层

很多 GitHub 上成熟的实践会单独放：

- `CONVENTIONS.md`
- `RULES.md`
- `AGENTS.md`
- `CLAUDE.md`

用于保存：

- 编码约定
- 工作边界
- 质量要求
- 与项目协作相关的固定规则

当前 skill 更强调“状态与记忆”，但对“项目规则层”的承接还偏弱。

### 4.4 缺少归档与体积控制机制

AAHP 和 handoff engine 很强调：

- 最近日志只保留最近 N 条
- 旧内容归档
- 索引文件长度受控
- 活跃任务数受控

当前 skill 虽然强调压缩，但还没有把这些约束写成更硬的规则，例如：

- `04-session-handoff.md` 最多保留最近 1 轮摘要
- `06-change-log.md` 超过一定长度要归档
- `05-open-questions.md` 只保留仍未解决的问题

这意味着后期仍可能出现文档膨胀。

### 4.5 缺少“失败沉淀层”

一些较成熟的 GitHub 实践会额外保存：

- 失败记录
- 改进日志
- lessons learned
- recurring mistakes

当前 skill 已经有“反模式”和“评估”，但项目运行时的失败经验还没有专门落位。

### 4.6 缺少更强的验证闭环

公开实践常见的更强版本会要求：

- 只有验证通过才写成已完成
- 变更记录必须带验证方式和结果
- 状态同步要和真实代码 / 文件事实交叉核对

当前 skill 已经有这个倾向，但还没有把它强化成更严格的执行约束。

## 5. 效果层面我目前“验证到什么程度”

### 已验证的部分

目前已经验证了：

- **结构完整性**：目录和文件职责清晰
- **入口逻辑**：`SKILL.md` 能清楚定义使用方式
- **初始化能力**：`scripts/init_context_pack.py` 可正常创建标准 `.ai-context/` 目录
- **方法论合理性**：与 GitHub 公共最佳实践的核心方向一致

### 尚未充分验证的部分

目前还 **没有** 做到以下这种“强验证”：

- 在多个真实项目里跑 1-2 周再复盘
- 做长会话前后对比，看是否明显减少遗忘
- 做多 agent 接力实验，看断层是否明显减少
- 做恢复开发计时，看是否真的更快
- 做文档膨胀测试，看 2 周后是否仍然可读

所以当前不能诚实地说：

> “效果已经被充分实证。”

更准确的表述应该是：

> **结构方向已经验证合理，外部最佳实践对齐度较高；但效果还需要结合真实项目连续使用来验证。**

## 6. 一个更客观的评分

如果按“与 GitHub 公共最佳实践对齐度”来粗略打分：

### 结构对齐度
- **7.5 / 10**

原因：

- 共享记忆、分层文件、决策日志、交接摘要、压缩理念都已经具备
- 但还缺 manifest / quick context / archive / lessons learned / stronger conventions layer

### 可落地程度
- **8 / 10**

原因：

- 已经足够用于真实项目启动和日常协作
- 初始化脚本也已经具备
- 不是概念稿，而是可以真的放进项目里用

### 实战验证程度
- **4.5 / 10**

原因：

- 目前主要是结构验证和方法论验证
- 还不是经过多个项目长期对照后的“成熟结论”

---

## 7. 最终结论

### 结论一句话版

**有对齐，而且方向是对的；但现在是“结构上已接近 GitHub 公开最佳实践的 v1”，还不是“效果已被充分验证的最终版”。**

### 更实用的判断

如果你现在就把它投入项目使用，**大概率已经能明显改善“AI 长对话后忘主线、隔天恢复慢、多人接力断层”这几个问题**。

但如果你想把它打磨成更接近 GitHub 上成熟上下文基础设施的版本，下一步最值得补的是：

1. 增加 `00-quick-context.md` 或 `MANIFEST.md`
2. 增加项目规则层（如 `07-conventions.md`）
3. 增加失败经验层（如 `08-lessons-learned.md`）
4. 增加归档和长度控制规则
5. 增加“验证通过后才能写完成”的硬约束

---

## 8. 推荐的下一版升级方向

建议把当前 skill 升级成 **v2**，目标不是大改，而是补齐以下增强项：

- **快速入口层**：`00-quick-context.md`
- **规则层**：`07-conventions.md`
- **经验层**：`08-lessons-learned.md`
- **归档规则**：给 handoff / change-log / open-questions 加长度控制与归档说明
- **更强验证闭环**：把“已验证 / 未验证”写进 change log 与 current state

这样它会更接近 GitHub 上成熟上下文工程实践里的“可持续版本”。
