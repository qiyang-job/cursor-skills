# AI 开工提问脚本

> 用途：在正式让 AI 写代码前，先用一组固定提问把问题、范围、最小闭环、改动面、风险与验收口径收敛清楚。  
> 适用场景：新功能立项、小版本迭代、规则新增、结构调整、怀疑需求范围过大时。  
> 配套文档：建议结合 `docs/Development-Harness.md`、`docs/开发前置模板.md`、`docs/收口决策模板.md` 一起使用。

---

## 1. 使用原则

这份脚本最重要的原则只有一句话：

### 先不要让 AI 直接写代码，先让它帮你把问题收敛清楚。

推荐使用方式：

1. 先复制对应脚本给 AI
2. 明确写上“**先不要写代码**”
3. 让 AI 先产出问题收敛结果
4. 你确认范围、边界与验收后，再进入实现

如果一开始就跳过收敛，直接让 AI 生成完整方案，通常会更快进入返工阶段。

---

## 2. 最常用的开场句

以下这句可以作为每次开工前的通用起手式：

```md
先不要写代码。请先帮我把这个需求收敛清楚：
1. 这次真正要解决的核心问题是什么
2. 最小可行闭环是什么
3. 这次明确不做什么
4. 主链路是什么
5. 改动面是否过大
6. 验收口径是什么
请优先帮我缩小范围，而不是补全一个完整系统。
```

如果你只想记一句话，记这句就够了。

## 3. 场景化脚本

## 3.1 开工前收敛脚本

适合：一个想法刚出现，还没决定是否要开发时。

```md
先不要写代码。请你先作为产品经理和架构师，帮我把这个需求收敛清楚。

请按下面结构回答：
1. 这次真正的核心问题是什么
2. 用户最痛的点是什么
3. 这次只应该聚焦哪一个问题
4. 最小可行闭环是什么
5. 这次明确不做什么
6. 主链路应该怎么定义
7. 如果现在就开发，最大的风险是什么
8. 有没有更小的实现路径

请优先帮我缩小范围，避免一开始就设计成完整系统。
```

---

## 3.2 最小闭环压缩脚本

适合：你已经有很多想法，但担心范围太大时。

```md
先不要写代码。请帮我把这个需求压缩成一个最小可行闭环。

请你输出：
1. 最小入口
2. 最短主链路
3. 最少必要状态
4. 最少必要数据结构
5. 最少必要交付结果
6. 最基本的验证方式
7. 这次可以明确先不做的部分

要求：
- 不要输出完整系统方案
- 不要顺手补太多未来能力
- 只保留第一轮必须跑通的部分
```

---

## 3.3 改动面审视脚本

适合：你担心一改就牵一大片时。

```md
先不要写代码。请先帮我判断这个需求的改动面是否过大。

请按下面结构分析：
1. 需要改动哪些模块 / 文件 / 层级
2. 是否会跨前端、后端、数据层、调度层联动
3. 是否会影响已有主链路
4. 是否会新增第二套规则源、状态判断或兜底逻辑
5. 这次改动面属于低 / 中 / 高，为什么
6. 如果改动面过大，请帮我把方案缩小成一个更安全的版本

目标不是做得更完整，而是让改动足够集中、可控。
```

---

## 3.4 验收口径定义脚本

适合：你担心“做完”的定义不清楚时。

```md
先不要写代码。请先帮我定义这次需求的验收口径。

请明确输出：
1. 什么叫做做完
2. 什么叫做跑通
3. 什么叫做稳定
4. 应该用什么方式验证
5. 哪些信号说明必须回炉
6. 哪些只是优化项，不应该阻塞本轮闭环

请避免模糊表达，尽量给出可观察、可验证的标准。
```

---

## 3.5 复杂度预警脚本

适合：你感觉“这次做着做着又要变重了”时。

```md
先不要写代码。请先帮我判断，这个需求现在更像是正常扩展，还是会明显增加系统复杂度。

请重点分析：
1. 会不会新增很多规则
2. 会不会让主链路变得更难解释
3. 会不会出现改一处动多处
4. 会不会增加上下文依赖和补丁式开发
5. 如果继续做，最可能恶化的点是什么
6. 有没有更小的替代方案

请优先站在控制复杂度的角度回答，而不是站在功能完整性的角度回答。
```

---

## 3.6 收口判断脚本

适合：你怀疑当前不该继续加功能，而该先收口时。

```md
先不要写代码。请先帮我判断，当前更应该继续开发，还是先暂停扩展做结构收口。

请按下面结构回答：
1. 当前系统最核心的主链路是什么
2. 当前最大的问题是功能缺失，还是结构耦合 / 规则分散 / 状态混乱
3. 如果现在继续扩展，最可能恶化的是什么
4. 如果先收口，最该先收哪一层
5. 这轮收口的最小目标是什么
6. 收口完成后，什么条件下才适合继续扩展

请不要急着给我实现方案，先帮助我判断方向。
```

## 4. 一键组合版脚本

如果你不想分场景挑脚本，可以直接用下面这份综合版。

```md
先不要写代码。请先帮我把这个需求收敛成一个最小、可执行、可验证的版本。

请按下面结构回答：
1. 这次真正的核心问题是什么
2. 用户最痛的点是什么
3. 最小可行闭环是什么
4. 这次明确不做什么
5. 主链路是什么
6. 需要改动哪些模块 / 文件 / 层级
7. 改动面是否过大，为什么
8. 最大风险是什么
9. 验收口径是什么
10. 如果这个方案还是太大，请帮我再缩小一版

要求：
- 先不要写代码
- 先不要设计完整系统
- 优先缩小范围
- 优先保证主链路清楚、改动集中、验收明确
```

## 5. 适合开发中的追问

当 AI 给出第一版分析后，可以继续追问：

- “请再缩小一版。”
- “请只保留第一轮必须跑通的部分。”
- “请明确写出这次不做什么。”
- “请指出最容易导致返工的设计。”
- “请判断这是不是提前抽象了。”
- “请只保留主链路，不要补边缘能力。”
- “请从控制复杂度而不是功能完整性的角度重答。”

这些追问的作用，是防止 AI 本能地把方案补成“看起来更完整”的版本。

---

## 6. 什么时候再进入写代码

只有当以下条件基本满足时，才建议切换到实现模式：

- 核心问题已经足够清楚
- 最小闭环已经明确
- 不做项已经写清楚
- 主链路能够简短说明
- 改动面可控
- 验收口径明确

然后再给 AI 下实现指令，例如：

```md
好，现在基于刚才收敛出的最小闭环开始实现。
要求：
1. 只实现本轮范围内的内容
2. 不要顺手扩展未来能力
3. 如果发现改动面超出预期，先停下来说明
4. 实现后按刚才定义的验收口径验证
```

---

## 7. 推荐搭配方式

最推荐的使用顺序如下：

1. 先读 `docs/Development-Harness.md`，统一原则
2. 再用本文件里的脚本和 AI 讨论
3. 讨论收敛后，填写 `docs/开发前置模板.md`
4. 真正开始开发
5. 如果中途感觉系统变重，再使用 `docs/收口决策模板.md`

---

## 8. 一句话提醒

### 如果你没有先告诉 AI“先不要写代码”，那它大概率会优先帮你补方案，而不是帮你缩范围。
