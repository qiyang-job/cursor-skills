# 真实案例与反例

> 目的：把这次项目里真实踩过的坑沉淀成可以反复复用的判断样例。  
> 用法：当 skill 需要判断“这是不是一个危险信号”“为什么这个方案虽然看起来能跑，但本质不稳”时，优先参考这里。

---

## 案例 1：一上来就让 AI 写代码，结果范围迅速膨胀

### 发生了什么

需求还没有把核心问题、最小闭环、不做项讲清楚，AI 就直接进入实现模式，顺手补了未来能力、补了边界、补了系统化结构。

### 暴露出的信号

- 第一轮改动面很快跨多层
- 方案越来越像完整系统，而不是最小闭环
- 用户需要不断把 AI 拉回“先做小一点”

### 反例教训

只要目标还不稳定，越快进入编码，返工通常越早到来。

### 正确抽象

skill 触发后，先做需求收敛，再允许进入实现。

---

## 案例 2：把聊天当成协作引擎，导致流程状态依赖语义猜测

### 发生了什么

产品层把“像真人一样聊天”作为主要体验，但系统层没有把流程控制、任务依赖、状态推进、续跑机制显式建出来，导致很多协作行为停留在消息层。

### 暴露出的信号

- 用户需要从消息内容里猜当前状态
- 任务衔接是否继续，很大程度依赖语义和上下文
- 异常恢复困难

### 反例教训

聊天可以是界面，不应该承担整个协作引擎的职责。

### 正确抽象

skill 在 AI 工作流场景下，必须区分：

- 消息流负责沟通、汇报、解释
- 流程流负责调度、依赖、状态和续跑

---

## 案例 3：把 Prompt 当成硬控制，导致主链路不稳定

### 发生了什么

系统试图依靠 Prompt 约束 agent 行为，承担任务衔接、完成判断、回退判断等关键控制逻辑。

### 暴露出的信号

- 一旦上下文变化，行为容易漂
- 结果是否继续推进不稳定
- 很多“应该能继续”的动作实际上没有稳定发生

### 反例教训

Prompt 可以做软约束，不能单独承担硬控制。

### 正确抽象

skill 在多步协作场景中，必须优先检查：

- 状态机是否显式存在
- 完成信号是否系统化
- 重试 / 回退 / 续跑是否可观测

---

## 案例 4：只跑到第一跳，后续任务没有被唤醒

### 发生了什么

队列里已经有后续步骤，但系统默认“既然排进队列，后面自然会继续”。实际上后续步骤没有被显式唤起，链路卡在中间。

### 暴露出的信号

- 第一跳成功，后续安静堆积
- 日志看起来有一部分动作，但主链路并未完整结束
- 用户以为是偶发，其实是机制缺失

### 反例教训

多步工作流里，不能默认排队就等于继续执行。

### 正确抽象

skill 需要显式检查“每一跳如何驱动下一跳”，而不是只看队列是否存在。

---

## 案例 5：base URL / 端口写死，导致本地与自动化链路脱节

### 发生了什么

本地服务运行在一个端口，自动调度或内部调用却打到了另一个默认端口，接口本身没问题，但工作流整体不成立。

### 暴露出的信号

- 页面层看起来正常
- 某些 API 单测也正常
- 但真实自动化链路就是不推进

### 反例教训

环境地址、端口、运行基座如果写死，最容易制造“局部正常、整体失效”的假象。

### 正确抽象

skill 在涉及内部 HTTP 调用、调度、续跑时，必须显式检查 base URL 是否被配置化管理。

---

## 案例 6：只看页面不报错，没有跑真实链路验证

### 发生了什么

开发中容易把“页面能打开”“静态渲染正常”“mock 返回正常”当成主链路成立的证据。

### 暴露出的信号

- UI 表面正确，但任务、数据库、调度状态没有真实验证
- 真创建项目、真发 brief、真跑一次后，才暴露出工作流断点

### 反例教训

不跑真实链路，就很容易高估系统成熟度。

### 正确抽象

skill 要推动至少一条真实链路验证：页面入口、API、状态变化、调度推进、最终完成信号。

---

## 案例 7：半迁移状态残留，最终在线上以运行时错误形式暴露

### 发生了什么

变量、旧入口、旧逻辑、旧默认值没有清干净，表面上功能已经迁到新路径，但渲染层还引用旧变量名，最终运行时报错。

### 暴露出的信号

- 新旧命名并存
- 页面改了一半、逻辑改了一半
- lint 不一定能拦住运行时引用错误

### 反例教训

半迁移状态是非常真实的高风险源，不能用“看起来差不多”来放过。

### 正确抽象

skill 在涉及收口、迁移或重构时，必须提醒：旧变量、旧入口、旧规则、旧默认值要么兼容完，要么清干净。

---

## 总结：这些案例对 skill 设计意味着什么

一个好的 kickoff skill 不应该只会讲理念，而应该具备以下能力：

- 在项目开始时，拦住过早编码
- 在 AI 工作流场景下，主动检查系统控制层风险
- 在项目变重时，帮助判断是否该先收口
- 在迁移或重构场景下，主动检查半迁移残留
- 在看似“已经差不多”的情况下，推动真实链路验证

一句话说：

### skill 的价值，不是让 AI 说得更漂亮，而是更早发现那些会在后面变成返工和故障的真实问题。
