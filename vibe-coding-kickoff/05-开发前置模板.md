# 开发前置模板

> 用途：在正式开发前，先把问题、范围、主链路、改动面、风险与验收口径收敛清楚。  
> 适用阶段：新功能立项、小版本迭代、较大改造、规则新增、结构调整前。  
> 配套原则：请结合 `docs/Development-Harness.md` 一起使用。

---

## 1. 使用方式

每次准备进入实现前，先完整填写这份模板。

如果以下任一部分无法清楚填写，建议先继续讨论，不直接进入开发：

- 核心问题不清楚
- 最小闭环不清楚
- 不做项不清楚
- 主链路不清楚
- 验收方式不清楚

这份模板的目标不是增加流程负担，而是减少后续返工和补丁式开发。

### 2.2 核心问题

- **这次要解决的核心问题是什么？**
- **这个问题现在为什么值得做？**
- **不做这件事，当前用户会卡在哪？**
- **这次只聚焦哪一个问题？**

> 要求：尽量只写一个主问题，不要把多个目标捆在一起。

---

### 2.3 目标用户与场景

- **谁会使用这项能力？**
- **发生在什么场景下？**
- **用户当前的操作路径是什么？**
- **这项能力插入到哪一步最有价值？**

---

### 2.4 本轮目标

- **本轮希望打通的最小闭环是什么？**
- **用户从入口到结果的最短路径是什么？**
- **本轮成功后，用户能多完成什么原本做不到的事？**

> 推荐写法：
>
> - 用户从 `A` 入口进入
> - 经过 `B` 主链路
> - 产生 `C` 结果
> - 用 `D` 方式验证结果是否成立

---

- **这次不处理的边界情况**：
- **这次不进入的系统层建设**：
- **以后再做的内容**：

> 要求：这一节必须显式填写，不能省略。

---

### 2.6 主链路说明

- **主入口**：
- **主状态流转**：
- **关键模块 / 页面 / 接口**：
- **最终产物 / 最终状态**：
- **失败时最重要的退路 / fallback**：

> 要求：这部分必须能用几句话讲清楚，不能依赖隐含上下文。

---

### 2.7 改动面评估

- **会改哪些文件 / 模块？**
- **这次改动是否跨前后端 / 数据层 / 调度层？**
- **是否会影响已有主链路？**
- **是否会新增第二套规则源 / 第二套状态判断 / 第二套兜底逻辑？**

#### 改动面判断

- **低**：局部修改，影响集中，职责清楚
- **中**：涉及多个模块，但边界仍清楚
- **高**：需要同时改很多层，且容易互相影响

如果改动面判断为 **高**，应先考虑缩小范围或先做结构收口。

---

### 2.8 风险与复杂度判断

- **最大的实现风险是什么？**
- **最大的产品风险是什么？**
- **最大的稳定性风险是什么？**
- **这次是否在引入新的规则复杂度？**
- **是否存在“看起来更完整，但会明显增加维护成本”的设计？**

---

### 2.9 验收口径

- **什么叫做做完？**
- **什么叫做跑通？**
- **什么叫做稳定？**
- **用什么方式验证？**
  - 页面操作
  - API 调用
  - 数据库状态
  - 日志 / 事件
  - 构建 / 类型检查
- **哪些信号说明必须回炉？**

- **问题定义是否清楚？** 是 / 否

- **改动面是否可控？** 是 / 否
- **验收口径是否清楚？** 是 / 否

## 4. 适合配合 AI 的使用方式

推荐先让 AI 参与以下事情，再让它写代码：

1. 帮你把需求压缩成一个核心问题
2. 帮你写出“这次不做什么”
3. 帮你找最小闭环
4. 帮你列出主链路与改动面
5. 帮你识别潜在高耦合点
6. 最后再让它开始实现

推荐提问方式：

- “请先不要写代码，帮我把这个需求收敛成最小闭环。”
- “请先帮我写出这轮明确不做什么。”
- “请先判断这个方案的改动面是否过大。”
- “请先帮我找更小的实现路径。”
