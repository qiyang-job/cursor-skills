# vibe coding 项目须知

> 目的：把这个文件夹做成 `vibe-coding-kickoff` 的**便携版 skill 包**。以后你可以把整个文件夹直接放进任意项目里，让 AI 先阅读这里的内容，再按这套规范开工，而不是一上来就写代码。

---

## 1. 这个文件夹现在是什么

这个文件夹不再只是几篇零散文档，而是一套可以直接复用的开工资料包。

它的目标不是让 AI “写得更快”，而是让 AI：

- **先收敛问题，再进入实现**
- **先做最小闭环，再谈完整系统**
- **先控制复杂度，再加智能化和自动化**
- **先跑真实链路，再相信系统真的成立**

一句话说：

### 这个文件夹的作用，是在项目开工前帮你守边界、缩范围、压闭环、定验收。

---

## 2. 以后怎么用这个文件夹

### 用法 A：直接把整个文件夹丢进项目里

推荐做法：把这个文件夹整个放进项目目录，例如 `docs/`、`playbook/`、`ai-context/` 或其他你常放规则文档的位置。

然后让 AI：

1. **先读 `SKILL.md`**
2. 再按 `SKILL.md` 里定义的规则，按需读取对应文档
3. 先做需求收敛、主链路判断、改动面评估、验收口径定义
4. 边界清楚后，再进入实现

### 用法 B：只想让 AI 快速理解

如果时间紧，至少先让 AI 读这几份：

- `SKILL.md`
- `01-AI开工总则（通用经验）.md`
- `02-新项目启动卡片.md`
- `04-AI-开工提问脚本.md`
- `05-开发前置模板.md`

### 用法 C：项目已经开始变重时

额外再让 AI 读：

- `03-Development-Harness.md`
- `06-收口决策模板.md`
- `07-真实案例与反例.md`

### 用法 D：想判断这套 skill 有没有效果

再补读：

- `08-效果评估与验收.md`

---

## 3. 最推荐的阅读顺序

建议以后每次都按这个顺序：

1. `SKILL.md`
2. `00-先读我.md`
3. `01-AI开工总则（通用经验）.md`
4. `02-新项目启动卡片.md`
5. `04-AI-开工提问脚本.md`
6. `05-开发前置模板.md`
7. 真正开始开发
8. 如果中途明显变重，再看 `03-Development-Harness.md` 和 `06-收口决策模板.md`
9. 如果要对照真实教训，再看 `07-真实案例与反例.md`
10. 如果要评估这套方法是否有效，再看 `08-效果评估与验收.md`

## 4. 给 AI 的推荐开场方式

你可以直接把下面这段复制给 AI：

```md
先不要写代码。

请先阅读这个文件夹里的 `SKILL.md`，并按其中的规则帮助我推进项目。
如果需要，请再按场景读取：
- 01-AI开工总则（通用经验）.md
- 02-新项目启动卡片.md
- 04-AI-开工提问脚本.md
- 05-开发前置模板.md

然后请先帮我完成以下事情：
1. 定义这次真正要解决的核心问题
2. 压缩出第一轮最小可行闭环
3. 明确写出这次不做什么
4. 说明主链路、关键状态和 fallback
5. 判断改动面是否过大
6. 明确验收口径

要求：
- 先不要写代码
- 先不要补完整系统
- 先优先缩小范围
- 先优先保证主链路真实可跑
- 如果发现方案会引入多层耦合、第二套规则源、状态混乱或半迁移状态，先停下来指出
```

## 6. 这包文档里各文件分别干什么

- `SKILL.md`：这套资料包的主入口，定义触发条件、工作流、输出契约和退出条件
- `01-AI开工总则（通用经验）.md`：最核心的通用经验总结
- `02-新项目启动卡片.md`：决定“现在该不该开工”的一页纸标准
- `03-Development-Harness.md`：更完整的方法论，约束项目如何避免越做越重
- `04-AI-开工提问脚本.md`：可直接复制给 AI 的收敛脚本
- `05-开发前置模板.md`：真正进入实现前要填的结构化模板
- `06-收口决策模板.md`：当系统开始变重时，用来判断是否先暂停扩展
- `07-真实案例与反例.md`：真实踩坑沉淀，用来帮助 AI 做更像工程判断的判断
- `08-效果评估与验收.md`：判断这套 skill 是否真的有效
- `09-Skill结构说明.md`：解释这整个便携包的组成与维护方式
- `scripts/generate_kickoff_bundle.py`：把结论生成成固定 Markdown 产物

---

## 7. 最后一句提醒

**AI 最容易帮你做快的，是代码；但一个项目最容易失控的，往往不是代码速度，而是范围、状态、规则和验证口径。**
